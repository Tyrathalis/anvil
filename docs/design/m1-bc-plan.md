# M1 plan — behavior cloning

**Date:** 2026-07-04. **Anchors:** design §1–3 (encoder, state, policy head), §6 phase 1 (BC + value), §9 (bridge, trajectory store); [ADR-0003](../decisions/ADR-0003-m0-closeout.md) (sequencing out of M0, ledgered caveats); [ADR-0004](../decisions/ADR-0004-m1-scope.md) (scope decisions this plan builds on); [bridge-protocol-v0.md](bridge-protocol-v0.md) (`observation` socket, `CastPlan` sketch); [callback-census-results.md](callback-census-results.md) (tag traffic on pool `f568b187`); [dc-pool-pipeline.md](dc-pool-pipeline.md) (pool `cf2ca6ba`, 1,701 cards / 113 decks).
**Question answered:** what M1 builds, in what order, and what number closes it — such that representations are validated with zero RL machinery (§13) and M2 inherits a working corpus + eval path, not a rewrite.

## The milestone in one paragraph

Generate a ~500K-game heuristic self-play corpus on the DC pool with full decision labels and entity-level observations logged at generation time (replay drift makes later re-derivation unreliable — ADR-0003 caveat), train encoder + trunk + policy/value heads by supervised imitation, and close on **held-out action agreement, measured honestly** (forced and pass-trivial decisions excluded, reported per-tag; §15 target: high-80s, P=80%). The eval path — real observations over the bridge + a `CastPlan` executor — makes the BC agent play complete games against the heuristic; winrate is recorded, not gated. The corpus is the expensive artifact, so everything that shapes it (observation schema, label extraction) lands and is pilot-validated before the full run.

## Settled decisions (recorded in ADR-0004, consequences here)

1. **The featurization line.** Java serializes a **versioned, structured, entity-level observation** (game objects with zone/controller/tapped/counters/attachments/damage/etc., game-agnostic record envelope per §1); **tensor assembly** (one-hots, dedup counting, embedding lookup) is a **deterministic Python transform**. Feature-engineering iteration is free; only state-extraction changes force regeneration. Refines §9's "featurize once, in Java" (written for the inference path); at inference the Python server runs the same transform before the GPU pass.
2. **Labels for everything, heads for the rung.** The worker logs options + chosen answer for **all** live callbacks (the census/provenance log already has the shape) plus observations. The scope cut is which model heads and eval realizers get built — extending BC to a new tag later is a new head on the existing corpus, no regeneration.
3. **Tag scope.** *Rung 1 (committed):* cast-composite priority (`CastPlan`) + the M0 one-field tags — mulligan (design-flagged as its own eval category), triggers, binary, number. *Rung 2 (stretch, droppable to the front of M2):* combat constructs (`AttackMap`/`BlockMap`). Ordering and replacement/static families are explicitly M2-era, in the census-measured order. Everything unbridged stays heuristic-fallback at eval, tagged — the winrate number measures a model+heuristic hybrid and the plan says so out loud. Covariate-shift note: narrow scope keeps eval games near the training distribution; widen only after the narrow version holds up.
4. **CastPlan is two deliverables.** The **label extractor** (serialize the heuristic's chosen SA — spell, targets, X, modes, folded optional costs — into a `CastPlan`-shaped label at logging time) is a **corpus-generation prerequisite**: census proved targets/X are injected onto the SA and never surface as callbacks, so an uninstrumented corpus has nothing for the pointer decoder to imitate. The **executor** (realize a model-proposed plan via the AI play path; the 65%-veto-rate fix) is eval-path only and lands later in M1. Payment stays engine-default per §3c.
5. **Embedding model.** Pin **Qwen3-Embedding-4B** (exact HF revision in the run manifest; Apache 2.0, top of open-weights MTEB mid-2026, selectable output dims). BGE-M3 is the alternate arm in a pilot-corpus ablation (nearest-neighbor sanity over card functions + downstream BC-agreement delta) — MTEB doesn't measure rules text, so the in-domain ablation decides. Sizing is irrelevant at our volume (~1,700 short texts, embedded once, cached fp16, never ships at inference); §1's distillation escape hatch covers ageing. Scryfall Tagger tags + LLM gap-filler **deferred** (scrape project; revisit if agreement plateaus — it's an ablation on the same corpus, not a regeneration).
6. **Corpus composition.** Meta-pool only, pool `cf2ca6ba`, both seats heuristic, personality randomized (§6), deterministic seeded deck pairs with every deck covered (census-driver pattern). Precon/low-power slices are a provenance-tagged minority ablation **later, if useful** (fundamentals curriculum, population breadth for the M2 belief head) — never an unmixable blend. Deck-power mix is not the §6 skill-conditioning mechanism (that's checkpoint Elo, later human data).
7. **Yield macros (§3b) move to M2** (amends ADR-0003's M1 line): BC has no stop-setting labels, and deriving pseudo-labels bakes heuristic judgment into exactly the held-instant decisions §3b protects. The `yield_directive` protocol hook stays reserved.

## Deliverables, in order

**D1 — Observation schema v1 + trajectory store v0.**
Versioned record: game-agnostic envelope (entity records, global-state record, decision context), Magic semantics as namespaced tags/fields (§1 hygiene). Schema version stamped in every record and in `run.json`. Java serializer in the fork (extends the census logger); Python reader + tensor-assembly transform in `anvil/encoder/`. Store v0: per-game zstd frames + index, provenance fields (source, engine hash, pool version, seed, personality, schema version) per §9, under `data/trajectories/<run>/`. Budget arithmetic: ~2–6KB/observation × ~1,136 callbacks/game ≈ ~4MB raw/game ≈ ~2TB raw at 500K games; near-identical consecutive observations should compress 10–30× → **~100–200GB target, inside §9's NVMe budget**. Keyframe+delta encoding is the fallback lever if the pilot measures worse. The corpus is regenerable (seeds + heuristic), so it needs no backup.

**D2 — CastPlan label extractor** (fork). Hook the AI's cast decision, serialize the chosen SA into the composite label. Census-informed specifics: optional costs fold in from the priority-evaluation window; modes may bind at either interception point; X/targets read off the SA. Sanity gate: extracted labels replay-validate against the logged game outcomes on a smoke batch.

**D3 — Pilot corpus: ~50K games (~2–3 nights).** Measures the three unknowns before the expensive run: bytes/game after compression, g/h with observation serialization on the game thread (haircut vs the ~2,300–2,900 g/h pool-adjusted heuristic rate), and label sanity at scale. Orchestrator-managed, overnight chunks (chunk 200, `nice 19`, detached launch, STOP-file at morning — all ledgered M0 conventions).

**D4 — Encoder + trunk + heads** (`anvil/encoder`, `anvil/state`, `anvil/policy`, `anvil/heads`).
Card encoder per §1: frozen Qwen3-Embedding-4B vectors (name + type line + cost + oracle text, instruction-formatted; fp16 cache) + structured features (pips, type one-hots, P/T, loyalty, keywords) + 32–64d ID embedding → 2-layer MLP fusion → ~256d. State per §2: entity tokens with Python-side multiset dedup + count features, global tokens, format-as-features, action-history window; **loop-detector feature omitted at M1** (no engine-side detection yet; census: 0 draws in 500 pool games). Trunk: 8–12 layers, d=512, 8 heads. Heads: autoregressive pointer decoder for rung 1 (action type → source → targets → modes/X), one-field heads, win-prob value head (§6 phase 1), turn-plan latent included (in base architecture, near-zero cost).

**D5 — BC training on the pilot + ablations.** Training loop (PyTorch, bf16, 4090), held-out splits: random-by-game (headline) and **held-out-deck-pairs** (generalization probe; early evidence for §15's zero-shot row). Embedding ablation (Qwen3 vs BGE-M3) decides the pin. Schema iteration happens here if anywhere — before the full run.

**D6 — Full corpus: 500K games.** ~22–28 overnight sessions (~a month calendar at nights-only; that's the planning assumption — daytime running is upside, not schedule). No rush; model work proceeds in parallel. Contingency if calendar chafes: corpus generation is CPU-only (no bridge, no GPU) and embarrassingly parallel — a rented multi-core VPS finishes it for tens of dollars; the orchestrator's chunk/manifest machinery already makes runs portable and mergeable by seed range. **Amended by [ADR-0006](../decisions/ADR-0006-500k-default-on-stopping-rule.md) (2026-07-06): generation is default-on behind fork hardening with a pre-registered stopping rule; 500K is an upper bound with an empirical exit, not a target (pair-space saturates ~50K — marginal games buy draw diversity, not matchups).**

**D7 — Train to the gate.** Headline metric: **held-out action agreement excluding single-legal-option decisions, reported per-tag**; for priority, report both raw and pass-excluded agreement (pass-dominance must not inflate the number). The single-legal-option exclusion is defined on the **timing-legal candidate basis** ([ADR-0005](../decisions/ADR-0005-d3-label-mask-semantics.md): the logged mask is a superset of expert-castable; payability unfiltered — marginally stricter than a payable-only basis). Value head reported as outcome log-loss/calibration, not gated. Target: high-80s on the honest metric (§15 row resolves against *this* definition).

**D8 — Eval path: BC agent plays.** Real observations over the bridge (the `observation: bytes` socket + the same tensor transform server-side), `CastPlan` executor with veto-rate telemetry (expect the 65% M0 rate to collapse — the model proposes targeted, costed plans), server-side batching across workers. Gates: complete games with fallback confined to unbridged tags; winrate vs heuristic recorded. Re-measure bridge tax with real payloads + inference (protocol doc anticipates this).

**D9 (stretch) — Combat constructs.** Labels already in the corpus (decision 2); heads + realizers only. Drops to the front of M2 if M1 runs long.

## Risks and open questions

- **Serialization overhead on the game thread** — unknown until D3; if ugly, move serialization off-thread behind a bounded queue (games are independent; ordering per game preserved).
- **Schema v1 misses something training wants** — mitigated by raw-ish records (most misses are transform-side, free); worst case is a ~month regeneration, which staging exists to avoid.
- **Qwen3 rules-text fit** — the D5 ablation decides; escape hatch designed.
- **CastPlan executor edge cases** (modes at two interception points, replacement effects mid-cast) — census mapped these; executor serves mode answers wherever asked, mid-resolution stays micro-step per ADR-0001.
- **Convoke/improvise callback loop** (2,788 calls in one census game) — could inflate corpus-generation wall on affected games; still logged as look-before-M2, watch it in D3.
- **Metric gaming residue** — even the honest metric can hide "agrees on easy boards"; per-tag breakdown + held-out-deck split are the checks. Pro-game agreement (§7) arrives later as the external anchor.

## M1 is done when

1. The honest held-out agreement number exists and is compared to the high-80s target (§15 row resolves, either way — a miss recalibrates, that's what the row is for).
2. The BC agent completes real games over the bridge with provenance-clean fallback (unbridged tags only) and a recorded winrate vs the heuristic.
3. The corpus + store are provenance-complete and M2 can consume them (V-trace start point, §6 phase 2).
