# Prior-Work Survey (pre-M0)

*2026-07-02. The survey the design doc (§12, §13) requires before M0. Organized by project, then by what it changes for Anvil. Web-researched; every load-bearing claim has a source link. Author-reported numbers are flagged as such.*

---

## 1. ygo-agent (Yu-Gi-Oh!, the closest analog)

**Repo:** [sbl1996/ygo-agent](https://github.com/sbl1996/ygo-agent). No paper; sources are README, `docs/`, source, the author's Zhihu posts (snippet-verified only — Zhihu blocks fetchers), and a [ygobbs2 forum thread](https://ygobbs2.com/t/%E6%B8%B8%E6%88%8F%E7%8E%8B%E5%BC%BA%E5%8C%96%E5%AD%A6%E4%B9%A0ai%E5%B1%95%E7%A4%BA%E4%B8%8E%E8%AE%AD%E7%BB%83%E5%8D%A1%E7%BB%84%E5%BE%81%E9%9B%86/487913).

**Shape:** ygopro-core (C++ rules engine) statically linked into an [envpool](https://github.com/sail-sg/envpool) C++ environment ([ygopro.h](https://github.com/sbl1996/ygo-agent/blob/main/ygoenv/ygoenv/ygopro/ygopro.h)) — engine compiled into the env process, **zero IPC**. The RL step boundary is engine decision points (`MSG_SELECT_*` messages), exactly the "engine emits a decision request, agent answers" pattern Forge's `PlayerController` implies. Training: Cleanba-style distributed actor-learner PPO in JAX ([cleanba.py](https://github.com/sbl1996/ygo-agent/blob/main/scripts/cleanba.py)), UPGO on by default, V-trace available as a flag, `gamma=1.0`, mirror self-play over a randomized deck pool. Started in PyTorch, switched to JAX (2024-04) for speed. Network is small: 2-layer transformer over entities (d=128) + LSTM (512) across steps + FiLM-scored action logits.

**Encoding:** fixed 160 zero-padded card slots × 41 feature bytes; player-perspective (imperfect info). **Card-ID embedding table initialized from LLM text embeddings** (shipped `embed.pkl`) — the same cold-start idea as Anvil's §1, already validated there.

**Action space — the big divergence from Anvil §3:** no pointer decoder. The *environment* serializes multi-card selections (targets, costs, tributes, chain order) into sequential single-pick RL steps via an env-side state machine ([action.md](https://github.com/sbl1996/ygo-agent/blob/main/docs/action.md)); the policy flat-scores the ≤24 materialized legal actions per step; LSTM memory carries intent across micro-steps. Legality masking is trivially exact because only legal actions are ever materialized. The author calls action-space design the hardest part of the project. Costs: more env steps per game decision; credit assignment smeared across micro-steps.

**Throughput (author-reported):** ~3K env steps/s on a 16-thread laptop; 20–30K steps/s on 128 cores (~200 steps/s/core, zero-IPC); ~1K SPS end-to-end training on a GTX 1650/16 envs. Flagship: ~100M games in 5 days on 32×4090s. Serious runs: "8×4090 + 128-core CPU for a few days."

**Deployment:** [ygoinf](https://github.com/sbl1996/ygo-agent/tree/main/ygoinf) FastAPI inference server + **TFLite export**; human-AI battles shipped in the Neos web client (2024-07). Wart: the featurizer is fully reimplemented in Python for the inference path — a duplication Anvil's featurize-once-in-Java design (§9) avoids by construction.

**Status/warts:** effectively unmaintained since **2024-08**; build broken on modern GCC/Clang ([issue #14](https://github.com/sbl1996/ygo-agent/issues/14)); use the fork [KohakuBlueleaf/M-YGO-Agent](https://github.com/KohakuBlueleaf/M-YGO-Agent) (fixes compilation; EDOPro backend broken, use YGOPro-v1). No rigorous human eval — replays posted to a forum for community judgment. Known blind spot: negation state under-featurized → occasional nonsense plays ("engine gaps become agent gaps"). Never delivered: Bo3/side-decking, deck building, league training — the parts Anvil considers core (Tutor, match play).

## 2. MageZero (MTG on XMage, the contrasting bet)

**Repo:** [WillWroble/MageZero](https://github.com/WillWroble/MageZero) (MIT), XMage fork at [WillWroble/mage](https://github.com/WillWroble/mage). **Actively maintained** (last push 2026-06-19); [v0.1.0-alpha](https://github.com/WillWroble/MageZero/releases) (2026-04) ships a precompiled XMage fork + `mz` CLI; `mz play` gives human-vs-agent in the vanilla XMage client. Sole author (Will Wroble), **explicitly open to collaboration** (Discord `discord.gg/R6pB6xuEy9`); his roadmap includes a general AI-vs-AI simulation framework "enabling MageZero and other MTG AI projects to scale evaluation and training" — directly adjacent to the cross-engine bridge idea (§12).

**Bet:** per-deck/per-matchup specialist agents with matchup-local action heads (~20 priority logits, ~60 target logits per matchup), explicitly to avoid generalist-scale resources. State: sparse binary features via **2M-slot hashed EmbeddingBag** (Weinberger feature hashing, ~200 active features/state) → single transformer encoder layer → heads. AlphaZero-style PUCT self-play (no Dirichlet noise; TD-blended value targets), **bootstrapped from XMage's built-in minimax AI**.

**Harness (most transferable part):** fork-don't-wrap — AI-player plugins live *inside* the engine; a Python Flask inference server (port 50052) serves the network to Java MCTS over sockets; HDF5 dataset handoff; `runner.py` orchestrates generations. Two-process pattern proven workable on a Java engine of Forge's complexity.

**Throughput (author-reported, the key numbers):** **~250 games/hour on 13 CPU threads** with a 300-sim MCTS budget (~18 s/game); ~20K games suffices per deck. **Parallelism is heap-bound, not CPU-bound: 150 sims/s single-thread → ~75 sims/s at 8 threads**, from game-state-copy allocation pressure; OOM mitigations (`-Xmx24g`, fewer games/generation) are in the setup guide. Roadmap target ~1K games/hour via lighter simulations.

**Results:** UWTempo 16% → 66% winrate vs. XMage minimax over a curriculum; author-estimated ~13 points below human pilots on those matchups. Signs-of-life creative lines reported. No standardized benchmark (author acknowledges none exists).

**Known simplifications:** autotap only (no mana-payment decisions — cf. Anvil §3c doing this deliberately with a consequential-choice override), no mulligans/trigger-ordering/replacement-effects in search, and **determinized search leaks hidden information** (both agents see future draws during search) — flagged by the author as fundamental, unfixed.

## 3. LearnForge post-mortem (RL-on-Forge, 2015–16)

**Repo:** [thesilencelies/LearnForge](https://github.com/thesilencelies/LearnForge) — Stephen Lilico's university fourth-year project, Oct 2015 – Jan 2016, one contributor, zero forks. His own post-mortem survives in his report repo ([mtg.tex](https://github.com/thesilencelies/4ypreport/blob/master/mtg.tex), [conclusions.tex](https://github.com/thesilencelies/4ypreport/blob/master/conclusions.tex)).

**What it was:** vendored the whole Forge tree, copied `forge-ai` into a `forge-learnedai` module, and overrode `PlayerController` to replace the heuristic state score with a small value network V(s) (MLP 160→200→200→1, experience replay, terminal ±10 reward) using a dead Java NN library he had to fork to keep building. Card pool restricted to vanilla creatures + basic removal.

**What killed it, in his words:** (1) **a memory leak in Forge under sustained headless self-play** throttled experience collection ("batches of experiences of sufficient size couldn't be gathered"); (2) **representation collapse** — max/sum pooling over an unbounded multiset of permanents "inherently carries a lot of data loss"; the agent's winrate never improved and its play looked random; (3) NaN/ReLU instability, abandoned mid-debug; (4) term-project deadline; the report's mainline pivoted to his other topic. Not maintainer hostility, not upstream churn.

**Why it reads as validation:** his proposed-but-never-built fixes are Anvil's architecture — set/entity encoders instead of pooling, parsing card text into features, V(s)+model over Q(s,a). The two actual killers (Forge memory behavior under self-play; entity representation) are respectively M0's job to measure and §2's job to solve.

**Also cataloged:** GitHub search finds *no other* RL-on-Forge attempt. The 2026 Gymnasium benchmark ([arXiv 2605.06066](https://arxiv.org/abs/2605.06066), 56-card pool) and everyone else built tiny custom envs instead of driving a real engine. [gcoter/magic-the-gathering-python](https://github.com/gcoter/magic-the-gathering-python) (the author of Forge issue #3756) reimplemented a stripped MTG in Python and reported the engine consumed most of the effort — the trade Anvil is refusing by paying the Forge-harnessing cost. Beware two fake sources in this space: tmikonen's "Deep AI for Magic" (April Fool's) and reinforcement-learning-for-mtg.com's PDF (LLM-generated, fictional authors).

## 4. Forge today (verified 2026-07-02)

- **The "maintainers interested in ML" claim is real and quotable:** the [wiki AI page](https://github.com/Card-Forge/forge/wiki/AI) says verbatim "If you want to train a model for the AI, please do. We would love to see something like that implemented in Forge," and confirms headless CLI self-play. In [issue #3756](https://github.com/Card-Forge/forge/issues/3756) (2023) maintainer tool4ever helpfully pointed an ML asker at the sim mode and `PlayerController` entry points. **Temper it:** interest is an open invitation, not an initiative — every ML/API issue to date (#3062, #4419, #4420) was auto-closed by the stale bot (~35 days) with nothing built. Design conversations must live in PRs or Discord, not issues.
- **The load-bearing warning** (contributor marthinwurer, [#3756](https://github.com/Card-Forge/forge/issues/3756)): "The main issue blocking a more robust tree-search AI… is having a more robust single-step system where the AI can copy the game state…, get a list of all possible choices…, and apply them… **The current simulation code is effectively a reimplementation of a good chunk of the engine, and isn't very stable.**" His [PR #3669](https://github.com/Card-Forge/forge/pull/3669) (merged 2023) restructured the main loop into per-step calls — a foothold, not a solution.
- **What exists for headless driving:** `SimulateMatch.java` (CLI `sim` mode, deck-vs-deck/tournament, headless-capable); `PlayerController` as the decision-point abstraction (the same hook LearnForge used in 2015 — still there); `forge-ai/simulation/` (`GameCopier`, `GameSimulator`, `GameStateEvaluator`, `SpellAbilityPicker`) as the fragile lookahead layer; even an in-repo ML precedent (`forge-lda` topic-model module). No daemon/server API — "headless" means batch CLI.
- **Repo health: excellent.** Pushed same-day as this scan; 354 contributors; **193 external PRs merged in June 2026 alone**; quarterly releases (2.0.13, June 2026); [CONTRIBUTING.md](https://github.com/Card-Forge/forge/blob/master/CONTRIBUTING.md) even has a pragmatic "Using AI coding agents" section (disclose agent-written PRs).

## 5. What this changes for Anvil

1. **The M0 number now has brackets.** ygoenv's zero-IPC C++ envs: ~200 steps/s/core. MageZero's Java engine with in-process plugin + socket inference: ~250 games/hour on 13 threads *with a 300-sim MCTS budget* (search-free rollouts should be much faster). Anvil's design target of 10+ games/min on 32 cores (§15: 60%) sits well inside what MageZero's numbers suggest is reachable *if* allocation pressure is controlled — which is the actual fight (see 2).
2. **Memory/allocation is the recurring killer, promote it.** LearnForge died to a Forge memory leak under sustained headless self-play (2016); MageZero's parallelism *halved* per-thread on heap pressure from game-state copying (2026). Two independent Java-engine data points. Concrete M0 additions: a **multi-hour soak test watching RSS/GC** as a first-class harness deliverable, and allocation profiling from day one — not after throughput disappoints (design §9 already lists it; this moves it up).
3. **State forking (§9) is riskier than the design doc assumed.** Forking is load-bearing for four systems (Mentor counterfactuals, ddmin, drills, pivotal-turn search), and Forge's `GameCopier`/simulation layer is described *by a Forge contributor* as an unstable reimplementation of a good chunk of the engine. Candidate design amendment: treat "fork fidelity" as its own differential-testable deliverable (fork-and-replay-no-op must reproduce the mainline trajectory) alongside the canonicalization patch — and expect the copy path, not gRPC, to be where the engineering time goes. This deserves an ADR once M0 sizes it.
4. **The pointer-decoder bet (§3) now has a named alternative.** ygo-agent serializes multi-part decisions into sequential single-pick env steps with flat scoring over ≤24 materialized legal actions + recurrent memory, and it worked at superhuman-ambition scale. Anvil's within-step autoregressive decoder is still justified (MTG's target/mode/payment structure is deeper, and the turn-plan latent wants a within-turn scratchpad), but M1 should keep the fallback in mind: if pointer decoding fights us, env-side serialization is proven. Also adopt their trick: **only materialize legal actions** — masking becomes construction, not filtering.
5. **The encoder bet gets external validation.** ygo-agent initializes card-ID embeddings from LLM text embeddings; MageZero's hashed sparse features are the cheap fallback if fusion ever needs one. Nobody has tested zero-shot generalization to unseen cards (§15's open 50%) — still genuinely open.
6. **Bootstrap recipe confirmed.** MageZero: engine's built-in heuristic as initial opponent pool, 16%→66%. Matches Anvil's Phase-1 heuristic self-play plan (§6); no change, just evidence.
7. **Community plan sharpens.** File the friendly Forge thread as a **PR-adjacent Discord conversation, not an issue** (stale bot). Cite the wiki's open invitation. MageZero's author is active, collaboration-friendly, and wants a cross-engine eval framework — the §12 cross-engine bridge has a willing counterparty; open contact early, cheaply.
8. **Both analogs skipped rigorous evaluation** (forum-judged replays; winrate vs. minimax with author-estimated human baselines). Anvil's eval battery (Ante certification, pro-agreement, exploiter, calibrated ladder) is genuinely differentiating — and worth keeping even under schedule pressure, because it's what neither predecessor can claim.
9. **Nobody has done RL-on-Forge but us and a 2016 term project.** The niche is real, the engine is healthy and welcoming, and the reasons prior work avoided it (engine-harnessing cost) are exactly the costs Anvil budgets for. LearnForge's failure modes are all specifically addressed by the current design; the one it adds is #2/#3 above.
