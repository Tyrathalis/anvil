# Survey skim: npiguet/price-predictor + WillWroble/MageZero

Skim tier, docs/specs only. Clones under the scratchpad (`price-predictor/`, `MageZero/`). "Evidenced" = read in a doc; "inferred" = my read between the lines.

## Top takeaways for Anvil

1. **price-predictor is a working Tutor prototype, end to end.** Scorer (Set Transformer + Bradley-Terry pairwise loss on match outcomes) → SA search over 23-spell decks → distilled one-shot picker (REINFORCE vs the frozen scorer) → self-play generations feeding the scorer back. Anvil's Tutor could lift this loop shape almost verbatim, with two upgrades already in hand: a vastly stronger pilot (his matches are Forge-AI-vs-Forge-AI; ours would be the BC/RL agent) and a vastly richer trajectory store (his per-game signal is one line of "cards cast / cards dead / winner / starter").
2. **His headline honest negative mirrors ours:** the picker out-optimizes SA on scorer-score (2.426 vs 2.36) but is *match-play parity* (52.3%, ~1.1σ, 4,442 Bo7 games) — "the frontier is now the scorer, not the policy" (`experiments/2026-05-22-gen5-picker-initial-training.md`). Same structure as our D3 "31% leak didn't cash out as winrate." Proxy-metric ceilings live in the reward model, not the policy — for Tutor, invest in the deck-scoring critic before the deck-building policy.
3. **Val accuracy ≠ deployed strength, measured brutally:** 2/3/4-layer scorers all ~72% val acc, but −20pp vs +7pp against Forge in real games (`diaries/2026-04-25`). His fix = rank models by match outcomes only. Validates Anvil's arms-as-standing-eval discipline; also his 12-pool evals flipped sign run-to-run (pool luck) — a small-n eval-noise cautionary tale that Ante directly addresses.
4. **Per-card "winnability" labels from self-play counters are a cheap, well-designed Tutor asset** (`specs/2026-05-03-card-winnability-pretraining.md`): 9 heads — score_play/score_draw (tempo-vs-reactive split), played_rate (dead-card disambiguation), cast_lift (casting effect vs ride-along-in-good-decks, his causal-attribution head), color_lift_WUBRG (deviation-framed synergy, diagonal collapses to zero) — with Bayesian shrinkage + per-head sample weights for low-n, plus a joint MLM aux loss (15% mask, weight 0.1) so long-tail cards still get per-token gradient. Anvil's store can compute strictly richer versions of all of these.
5. **His deterministic-features-crowd-out-text finding is our ID-crowds-out-text finding** (`experiments/2026-05-02-deterministic-feature-reliance.md`, summarized in the winnability spec): the 32 hand features carried deck *shape*, the 512 text dims were supposed to carry card *quality* but were pretrained on a misaligned signal (prices, confounded by reserved-list/collector effects). His remedy = retrain the encoder on play-derived targets. Same lesson as ADR-0007's shortcut-learning note, converged independently.
6. **Reward-hacking audit pattern worth stealing for any Anvil component trained against a frozen learned scorer:** `train-picker --auditor-scorer-checkpoint` runs a per-epoch cross-scorer Spearman audit on val decks — a second scorer as canary for policy-overfits-the-reward-model. Cheap, structural, fits our "engine adjudicates every claim" invariant.
7. **MageZero is the cautionary contrast on state fidelity and hidden info.** Their MCTS searches with `see_opponent_hand: true` and fully determinized futures — full information leakage, acknowledged as a "fundamental challenge" and only "partially mitigated through discount factors" (`faq_goals.md`). Nothing anywhere on state-copy fidelity or determinism regression testing (that code lives in their XMage fork, undocumented here). Anvil's forkcheck/twin-determinism/info-set-transform work is precisely the layer they skipped; their throughput (~250 games/h at 13 threads, 24 GB heaps, ≤6 parallel games recommended) vs our ~1,146 g/h at 2 GB heaps is the price.
8. **Two small MageZero ideas worth noting:** (a) opponent-policy head trained from the agent's own MCTS visit counts + K virtual visits on the observed opponent action — a cheap opponent model that might interest D6+ self-play someday; (b) MuZero-style TD-λ blend over search root values as value targets (λ 0.9→0.7 annealed per generation) — relevant if Anvil ever adds search; not now.
9. **Both projects independently landed on Anvil's harness shape** — Python supervisor + disposable Java workers + append-only files + restart-on-crash + sentinel-prefixed stdout lines. price-predictor's zero-infrastructure 3-machine fan-out (NAS ZFS/SMB + Docker + hostname-sharded output files because SMB appends aren't atomic; reader globs shards and merges by timestamp) is a ready-made recipe if Anvil ever needs a second box for generation.
10. **Draft-agent live integration has one gating test Anvil should remember for any future online-state path:** online incrementally-reconstructed state must be *byte-equal* to what the offline trainer builds from the finished record, verified by replay (`specs/2026-06-04-draft-agent-live-play.md` §3) — the same serve/loader-parity philosophy as our `test_serve_parity.py`. Also his corpus-purity rule: any fault in a model seat's pick **abandons the whole draft** — never substitute a pick, never let degraded data into the self-play corpus.

## Repo 1: npiguet/price-predictor

Despite the name: a card-price predictor that grew into an MTG sealed-deck ML stack on Forge data. Heavily documented — speckit specs (`specs/0NN-*/`), dated prose specs (`specs/2026-*.md`), experiment writeups (`experiments/`), daily diaries (`diaries/`). Solo dev + Claude, Feb–Jun 2026.

### (a) Card embeddings (evidenced)
- **Corpus:** a Java `forge-connector` batch-converts all ~32K Forge card scripts into lowercase "LLM-friendly" text (`name:/mana cost:/types:/spell[1]:` lines). Custom word-level MTG tokenizer, ~5K vocab (spec 010).
- **Encoder v1 (price):** d=256 transformer trained on log-EUR prices; card vector = `cat([max_pool, mean_pool])` → 512d, +32 deterministic features (mana, types, colors, P/T, mana production) → 544d per card, cached as one `.npz` per card.
- **Encoder v2 (winnability, spec 016 / `2026-05-03`):** trained **from scratch** (price init deliberately rejected as bias re-introduction) on per-card targets aggregated from `cards-played.txt` — one line per game: cards cast vs dead per side, winner, starter. Stats per card: wins/losses × played/in-deck × play/draw × with-color-X counters. The 9 regression heads + MLM as in takeaway #4; heads are deliberately bare `Linear(d_card, …)` ("if a linear projection can't fit it, the encoder is the problem"). `name:` line stripped to block name→label shortcut. Card-level stratified train/val split.
- **Scale:** ~974K games at gen-5 time (`experiments/2026-05-11` — the "1M games" claim checks out), from ~70K Bo7 matches.

### (b) Deck scorer + SA + distilled picker (evidenced)
- **Scorer:** Set Transformer (SAB layers + PMA pooling + MLP head) over the deck's 544d card vectors; each match → pairwise (winner, loser) example, Bradley-Terry BCE on score difference; optional margin weighting (4-0 sweeps > 4-3). ~72% val acc ceiling, diagnosed as label entropy (43% of Bo7s genuinely close). Depth: 6 SAB layers won at scorer scale; gen-4 512d scorer beat `forge-best` 69.4% head-to-head (n=1,448).
- **SA search:** per pool, start from random (or color-pair-seeded) 23-spell deck, single-card swaps scored by the frozen scorer, T=0.8, cooling 0.85, ≤200 iters, restarts. +10.8% over pure greedy; the diary records *why* greedy fails (single swaps can't execute the coordinated multi-card move of dropping a color).
- **Picker (spec 017):** one-shot policy transformer, one logit per pool card, deck read off by a deterministic quota walk; trained by REINFORCE (64 sampled decks/pool, entropy annealing, aux pool-quality MSE head) then a reward-ranked top-k objective chained over 4 runs. Framed explicitly as an **amortized optimizer** replacing tens of thousands of scorer calls with one forward. Outcome: takeaway #2.

### (c) Draft agent + live Forge integration (evidenced)
- **Gen-1 (spec `2026-05-28`):** two-headed set transformer. State = typed token sequence: `[CONTEXT] [POOL] [PACK] [PASSED] [TAKEN]`; each card token = `.npz` vector + 4-dim type one-hot + two recency embeddings (`packs_ago`, `pick_ago`); `PASSED→TAKEN` via wheel-diff and pack-end flush (opponent-knowledge tracking from the seat's own observations only). Policy = shared linear over `PACK` tokens, masked softmax; critic = linear on `CONTEXT` predicting leave-one-out pod-relative deck score (final pools built by the frozen picker, scored by the frozen scorer — learned components labeling learned components, three frozen models deep). Corpus from Forge's draft AI with noise seats (`forge-r30`/`forge-r100` random-override agents) for state diversity; imitation whitelisted to `forge-full` seats, critic trains on all.
- **Results (`experiments/2026-05-30`, appended through 06-05):** LR decay ladder 3e-4→3e-6 is the whole game (top1 0.79→0.870); 6 layers beats 4 (top1 0.877); **8 layers regresses the policy head** (top1 −1.9pp) and the writeup pins it as an *optimization* failure, not overfitting — train loss is worse too, and 8-layer led at high LR then lost every decay ("the annealing tuned for 6 layers leaves 8 in a worse basin").
- **Live integration (spec `2026-06-04`):** worker↔supervisor stdin/stdout sentinel-line protocol (`<<DRAFT-PICK-REQUEST>>`/`RESPONSE`/`EVENT-JSON`), strict synchrony (≤1 outstanding request, no threading, no deadlock by construction); request carries only the pack in hand — Python reconstructs everything else incrementally, gated by the online≡offline state-equality replay test; any fault drops the whole draft (no substitute picks, corpus purity by construction). Any pod seat subset model-piloted; same `drafts.jsonl` doubles as self-play corpus and in-pod strength measurement (shared boosters cancel pool luck).

### (d) Match generation at scale (evidenced)
Python supervisor spawns N Java `MatchWorkerMain` processes (fat JAR + Forge jars); each picks a random sealed set, generates two 6-booster pools, builds decks by 4 weighted Forge methods (40/30/20/10, optimal→random — deliberate quality spread for scorer signal), plays Bo7, appends one line; supervisor restarts crashed workers. Self-play arms sample decks from prior generations' `generated-decks.txt` files with weights. 3-machine fan-out per takeaway #9 (`specs/2026-04-23`): no coordinator, hostname-sharded append-only files, Docker image with Forge pinned by SHA build-arg. Workers: 8 (NAS Ryzen 9700X) + 6 (desktop) + 2-4 (laptop).

### (e) Tutor relevance (inferred)
Takeaways #1–6. Nearest-term steals: the counter-based per-card label design (compute from our store, strictly richer inputs available — per-decision context, life swings, Ante-corrected outcomes instead of raw wins), Bradley-Terry pairwise deck scoring on paired-seed match outcomes, the amortized-optimizer framing (search-labeled decks → distilled one-shot builder), the auditor-scorer audit. His whole stack scores *decks as card multisets* with no play data in the deck representation — Anvil's pilot-in-the-loop lets Tutor label decks by actual piloted winrate, which is exactly the ceiling he hit and couldn't break.

## Repo 2: WillWroble/MageZero

AlphaZero-style deck-local RL on XMage. Small repo (~34 files): README, `faq_goals.md`, `setup_guide.md`, configs, ~10 Python files; **all engine-side substance (state encoder, MCTS, state copying) lives in his XMage fork**, not here. Solo dev, ~1 year in, v0.1.0-alpha released; active Discord.

### (a) "Each deck is its own game" (evidenced)
Framework trains one agent per deck ("deck-local"): the priority policy head's ~20 logits map to *that deck's* castable spells/abilities; a matchup-local target head (~60 logits) and a binary head cover micro-decisions; a separate opponent-priority head predicts the opponent's actions (both MCTS agents share one net in mirror training). State: Weinberger feature hashing into a 2M-slot sparse space shared across all decks (a 60-card matchup touches ~5K slots, ~200 active per state, collision <0.01%); hierarchical sub-feature pooling (ability→card→"green permanents count") for redundancy. Net: 2M×512 sparse embedding → **single** TransformerEncoderLayer (d=512, 4 heads) → mean pool → 256 hidden → heads; 30% input-token dropout; SparseAdam+Adam. The transformer replaced an EmbeddingBag-sum that destroyed co-occurrence and caused systematic misattribution (their "Skrelv problem"). Mixed-batch training with per-sample policy-head gating by decision type.

### (b) MCTS / state copy / fidelity (evidenced-by-absence)
PUCT c=1.0, no Dirichlet noise or temperature ("MTG already has a lot of inherent randomness" + deeper search instead). How game state is forked per simulation is **not documented in this repo** (pointer to fork modules `Player.AI.MCTS`, `Mage.MageZero` only); zero discussion of copy fidelity, divergence, or determinism regression anywhere — no analogue of forkcheck exists. The costs show up as throughput: single-thread ~150 sims/s, 8-thread only ~75 (heap-bound), ~18 s/game at full budget, ~250 games/h at 13 threads, `-Xmx24g` default, ≤6 parallel games recommended, "Optimize search — I've done all I can I fear; need help from experienced java/XMage engine devs." Their #5 future goal is ~1,000 games/h — below Anvil's already-measured 1,146 g/h with a model in the loop.

### (c) Training loop (evidenced)
`mz train` curriculum: per generation → self-play vs opponent pool (minimax or MCTS) → HDF5 → dataset stats → eval prev model → train (50 epochs bootstrap / 10 from checkpoint). Configs: 5–6 generations × 100–200 games/opponent, replay buffer ~10 generations; ~20K MCTS games "enough for most decks"; prior heads activate on a schedule (binary gen2, priority gen3, target gen4); value target = MuZero-style TD-λ blend over MCTS root scores, λ annealed 0.95→0.70 across generations. Results (Dec 2025): UWTempo 16%→66% vs 5 minimax mono decks; avg 47.9% vs an estimated-human 61% against the same bots (honest −13pp gap self-reported). Hardware: one desktop, local flask inference server ~100 inferences/s, HDF5 files moved manually between generations (async replay buffer is a stated goal).

### (d) Hidden information (evidenced)
Punted, candidly: `game.yml` ships `hidden_info: see_opponent_hand: true`; the FAQ states training games are deterministic and both agents search with leakage of future draws, opponent hands, and random outcomes, "partially mitigated through discount factors" — a fundamental limitation they defer to future MuZero-style dynamics models. The *network's* state representation excludes hidden info; only the search leaks. Also simplified: autotapper-only mana payment, no mulligans, no trigger ordering, no sideboarding, max_turns 50.

### (e) Steal / avoid (inferred)
**Avoid:** determinized-leaky search as a training environment (their emergent-behavior anecdotes — e.g. the genuinely clever delayed-Cecil line — are confounded by the agent having seen the opponent's hand); undocumented state-copy fidelity (ADR-0002 showed what actually lurks there in a mature engine). **Steal/watch:** the observed-action virtual-visit opponent-policy blend (`Ñ(s,a) = N_pred + k·[a=obs]`) as a near-free opponent model; input-token dropout as regularizer (echoes our planned ID dropout); the deck-local framing as a *Grindstone* thought — per-matchup specialist heads/adapters for drilling, not as a pilot architecture. Their FAQ confirms the ecosystem gap Anvil's fork API targets: "is there a python gym environment or API for AI in XMage? no... all AI projects use their own fork + local server."
